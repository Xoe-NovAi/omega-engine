## Purpose
To prevent "architectural amnesia." Before implementing any new feature, the agent must mine the legacy repositories (`xna-omega-legacy/` and `omega-stack-legacy/`) to reclaim proven patterns, schemas, and logic that were previously developed.

## The 5-Step Mining Workflow

### 1. Keyword Expansion
Define the core concept (e.g., "soul evolution"). Expand this into a set of search terms:
- Synonyms: `experience extraction`, `lesson abstraction`, `soul growth`.
- Technical terms: `jsonl`, `summarization`, `vector_store`, `metadata_tagging`.

### 2. Broad Search (The Net)
Use `grep` or the `knowledge-miner` skill to scan legacy directories.
- Search for the expanded keywords.
- Identify files that contain implementation logic (Python, JS) or strategic specs (Markdown).

### 3. Pattern Extraction (The Sift)
Read the identified files. Extract:
- **Logic**: How was the problem solved? (e.g., "Used a recursive summary loop").
- **Schema**: What was the data structure? (e.g., "A list of objects with `trace_id` and `lesson`").
- **Failure Points**: Why was this abandoned or pivoted? (Check `PIVOT_LOG.md` if available).

### 4. Gap Analysis (The Bridge)
Compare the legacy pattern with the current Omega Engine architecture.
- Does the legacy pattern fit the "Local-First" (Ryzen 5700U) constraint?
- Does it align with the current `EntityRegistry` (YAML-based) instead of PostgreSQL?
- What needs to be updated to make it "Sovereign"?

### 5. Reclamation Report (The Record)
Produce a short report in `docs/research/` following the Universal Recording Protocol.
- **Pattern Name**: (e.g., "Recursive Lesson Abstraction")
- **Source**: (File path in legacy repo)
- **Reclaimed Logic**: (The core algorithm/schema)
- **Integration Plan**: (How it will be implemented in the current Engine)

## Execution Workflow

3. **Mine**: Execute steps 1-5.
4. **Record**: Write the Reclamation Report.
