## §0 Purpose

The Jem 2.0 Split Test Framework evaluates **four architectural variants** (A-D) of the speculative decoding research pipeline. Each variant assigns different models to the Tier 2 (Reviewer) and Tier 3 (Synthesizer) roles, while Tier 1 (Jem's Draft) consistently uses **Gemma 4 31B**.

The goal: measure which model assignment strategy produces the highest-quality research outputs, richest training signals, and most efficient pipeline.

---

## §1 Variant Definition

| Variant | T1: Draft (Jem) | T2: Reviewer | T3: Synthesizer | Hypothesis |
|---------|----------------|--------------|-----------------|------------|
| **A (Control)** | Gemma 4 31B | Gemma 4 31B | Gemma 4 31B | Different prompts suffice — "Council of One" |
| **B (Frontier-T2)** | Gemma 4 31B | **DeepSeek V4 Flash** | Gemma 4 31B | Frontier reviewer catches more errors |
| **C (Frontier-T3)** | Gemma 4 31B | Gemma 4 31B | **DeepSeek V4 Flash** | Frontier synthesis adds strategic depth |
| **D (Full Frontier)** | Gemma 4 31B | **DeepSeek V4 Flash** | **DeepSeek V4 Flash** | Maximum quality ceiling |

## §2 Metrics Registry

### 2.1 Primary Metrics

| Metric | Definition | Collection Method | Target |
|--------|-----------|-----------------|--------|
| **Q-Score Δ** | Reviewer's quality score after - before | Parsed from Reviewer output | > +2.0 |
| **Acceptance Rate** | % of T1 drafts accepted on first pass | Quality Gate counter | > 60% |
| **Correction Count** | Number of corrections per run | Parsed from Reviewer output | — (measured) |
| **Correction Delta** | Character diff between draft and verified text | Levenshtein ratio | — (measured) |

### 2.2 Secondary Metrics

| Metric | Definition | Collection Method |
|--------|-----------|-----------------|
| **Gap Detection Rate** | Gaps identified per run | Parsed from T2/T3 output |
| **New Tasks Created** | Research tasks generated by T3 | Parsed from T3 output |
| **Tier 2 Latency** | Wall-clock time for review | Timing decorator |
| **Tier 3 Latency** | Wall-clock time for synthesis | Timing decorator |
| **Total Pipeline Time** | End-to-end wall clock | Timing decorator |
| **Token Cost T1** | Tokens consumed by Jem's draft | ModelGateway response |
| **Token Cost T2** | Tokens consumed by Reviewer | ModelGateway response |

### 2.3 Derived Metrics

| Metric | Formula | Purpose |
|--------|---------|---------|
| **Efficiency Ratio** | `Q-Score Δ / Total Pipeline Time` | Quality per second |
| **Token Efficiency** | `Q-Score Δ / Total Tokens` | Quality per token |
| **Training Signal Density** | `Correction Delta / Draft Length` | How much Jem needs to change |
| **Reviewer Value Add** | `Q-Score Δ (Variant B) - Q-Score Δ (Variant A)` | Incremental value of frontier |

## §4 Runbook

### 4.1 Initial Baseline (First 20 runs)

```
Phase 1: Baseline → Run variant A (Control) 20 times
  Goal: Establish baseline metrics for "Council of One"
  Minimum: 10 unique research prompts
  Output: Baseline metrics table
  
Phase 2: Exploration → Run variants B-D for 10 runs each (30 total)
  Goal: Measure quality deltas from baseline
  Minimum: 5 unique research prompts per variant
  Output: Variant comparison table
  
Phase 3: Analysis → Statistical significance testing
  Goal: Determine optimal variant
  Method: Paired t-test on Q-Score Δ, Acceptance Rate
  Output: Optimal variant recommendation
```

### 4.2 Ongoing Operations (After Baseline)

```
Each background cycle → Assign variant round-robin (A→B→C→D→A)
Daily → Generate aggregate report
Weekly → Compare variant performance
Bi-weekly → Rebalance model assignments
Monthly → Retrain Jem on accumulated training data
```

### 4.3 Manual Override

```bash
# Force specific variant for a run
omega jem run "research question" --variant B

# Run all 4 variants on same prompt (for direct comparison)
omega jem test run "research question"

# Generate report
omega jem test report
```

## §6 Statistical Analysis Method

### 6.1 Significance Testing

After Phase 2 (50 runs total):

```python
from scipy import stats

# Compare Variant B vs A for Q-Score Δ
t_stat, p_value = stats.ttest_ind(
    variant_b_q_deltas,  # array of Q-Score Δ for B runs
    variant_a_q_deltas,  # array of Q-Score Δ for A runs
)

# If p < 0.05: Variant B is statistically significantly better
# If p ≥ 0.05: Keep running; insufficient evidence
```

### 6.2 Variance Analysis

```python
# Check if frontier models introduce more variance
import numpy as np

b_variance = np.var(variant_b_q_deltas)
a_variance = np.var(variant_a_q_deltas)

# If b_variance > a_variance * 1.5: frontier model is inconsistent
# Consider: more runs needed, or frontier model unreliability
```

## §7 Escalation Path

| Condition | Action |
|-----------|--------|
| Any variant shows ≥ 25% regression vs control | Disable that variant, flag for Architect review |
| Tier 2 latency > 120s consistently | Switch to smaller/cheaper frontier model for review |
| Training data quality = consistently corrections contain "I agree" | Reviewer prompt needs hardening |
| False acceptance rate > 5% (human audit) | Tighten Quality Gate, harden Reviewer prompt |
| All variants underperform vs manual research | Pause pipeline, redesign architecture |
